# Mattilsynet Design Kotlin + HTMX

Kotlin/JVM helpers for rendering `@mattilsynet/design` HTML with [`kotlinx-html`](https://github.com/Kotlin/kotlinx.html), plus small generic HTMX attribute helpers.

The design system source of truth stays in the existing JS/TS/CSS package. This Gradle project consumes the root `mtds/` build output, including generated `mtds/component-metadata/*.component.json` files, to generate most Kotlin component wrappers during the build.

## Prerequisites

From the repository root you need:

- Node/npm compatible with the root `package.json` engines.
- A JDK available for Gradle/Kotlin.
- npm dependencies installed:

```sh
npm install
```

## Quick start from a fresh checkout

Run all commands from the repository root:

```sh
npm install
npm run build
./gradlew -p kotlin build
```

What this does:

1. `npm install` installs the JS build tooling.
2. `npm run build` creates the `mtds/` directory with CSS, JS, icons, illustrations, `styles.json`, and generated component metadata.
3. `./gradlew -p kotlin build` generates Kotlin wrappers from component metadata, copies resources into the jar, compiles, and runs tests.

## Develop the Kotlin package

Most Kotlin component code is generated. Do not edit files under `kotlin/build/generated/...` by hand.

The `mtds/component-metadata/*.component.json` files are generated by the npm build, not by the Kotlin build. The checked-in source is the TS component contract next to the JS/TS component source, for example `designsystem/button/button.contract.ts`. Update those contracts when the server-rendered HTML surface for a component changes.

When changing the Kotlin package:

1. Update stable Kotlin infrastructure when needed:
   - `kotlin/src/main/kotlin/io/mattilsynet/design/Mtds.kt`
   - `kotlin/src/main/kotlin/io/mattilsynet/design/htmx/Htmx.kt`
2. Update component contracts when adding or changing simple wrappers:
   - `designsystem/<component>/<component>.contract.ts`
3. Regenerate and test:

```sh
./gradlew -p kotlin clean build
```

Useful focused commands:

```sh
# Regenerate component metadata from TS contracts only
npm run build:metadata

# Generate Kotlin wrappers and class map only
./gradlew -p kotlin generateMtdsKotlin

# Copy mtds resources into the generated resources directory
./gradlew -p kotlin copyMtdsResources

# Run only Kotlin tests
./gradlew -p kotlin test

# Build jar without publishing
./gradlew -p kotlin jar
```

If the root JS/CSS build or component contracts changed, run `npm run build` before the Kotlin build so the generated Kotlin class map, component wrappers, and packaged resources use the latest `mtds/` output.

## Build and publish locally

For local testing in another Kotlin application:

```sh
npm run build
./gradlew -p kotlin publishToMavenLocal
```

This publishes:

```txt
io.mattilsynet:design-htmx-kotlin:<version-from-package.json>
```

The version comes from the root `package.json`.

Then in a consuming Kotlin app:

```kotlin
repositories {
    mavenLocal()
    mavenCentral()
}

dependencies {
    implementation("io.mattilsynet:design-htmx-kotlin:<local-version>")
}
```

For active local development, a consuming app can use a composite build instead of publishing after every change:

```kotlin
// settings.gradle.kts in the consuming app
includeBuild("../path/to/design/kotlin")
```

## Run the Kotlin code

This repository currently provides a library, not a standalone Kotlin server application. The normal ways to run it are:

1. Run the Kotlin test suite:

```sh
./gradlew -p kotlin test
```

2. Publish or include the library in a consuming server-rendered Kotlin app and run that app.

### Minimal Ktor usage example

In a consuming Ktor app, add the dependency as shown above, then render HTML with `kotlinx-html`:

```kotlin
import io.ktor.server.application.call
import io.ktor.server.html.respondHtml
import io.ktor.server.http.content.staticResources
import io.ktor.server.routing.get
import io.ktor.server.routing.post
import io.ktor.server.routing.routing
import io.mattilsynet.design.html.Alert
import io.mattilsynet.design.html.AlertColor
import io.mattilsynet.design.html.Button
import io.mattilsynet.design.html.ButtonVariant
import io.mattilsynet.design.htmx.hxPost
import io.mattilsynet.design.htmx.hxTarget
import kotlinx.html.body
import kotlinx.html.div
import kotlinx.html.head
import kotlinx.html.id
import kotlinx.html.link
import kotlinx.html.script

routing {
    staticResources("/mtds", "META-INF/resources/mtds")

    get("/") {
        call.respondHtml {
            head {
                link(rel = "stylesheet", href = "/mtds/styles.css")
                script(src = "https://unpkg.com/htmx.org@2.0.4") {}
                script(src = "/mtds/index.iife.js") {}
            }
            body {
                Button(
                    variant = ButtonVariant.Primary,
                    attrs = {
                        hxPost("/save")
                        hxTarget("#result")
                    },
                ) {
                    +"Lagre"
                }

                div {
                    id = "result"
                }
            }
        }
    }

    post("/save") {
        call.respondHtml {
            body {
                Alert(color = AlertColor.Success) {
                    +"Lagret"
                }
            }
        }
    }
}
```

## Usage in kotlinx-html

```kotlin
import io.mattilsynet.design.html.Alert
import io.mattilsynet.design.html.AlertColor
import io.mattilsynet.design.html.Button
import io.mattilsynet.design.html.ButtonVariant
import io.mattilsynet.design.htmx.hxPost
import io.mattilsynet.design.htmx.hxTarget

Button(
    variant = ButtonVariant.Primary,
    attrs = {
        hxPost("/lagre")
        hxTarget("#resultat")
    },
) {
    +"Lagre"
}

Alert(color = AlertColor.Warning) {
    +"Noe gikk galt"
}
```

Link-style buttons are separate from button elements:

```kotlin
import io.mattilsynet.design.html.ButtonLink
import io.mattilsynet.design.html.ButtonVariant

ButtonLink(href = "/dyr/123", variant = ButtonVariant.Secondary) {
    +"Se detaljer"
}
```

## Static resources

The jar includes selected static resources under `META-INF/resources/mtds` and the copied class map under `mattilsynet-design/styles.json`.

Ktor example:

```kotlin
routing {
    staticResources("/mtds", "META-INF/resources/mtds")
}
```

HTML example:

```kotlin
head {
    link(rel = "stylesheet", href = "/mtds/styles.css")
    script(src = "/mtds/index.iife.js") {}
}
```

The package does **not** bundle HTMX. Applications should include HTMX through their preferred frontend asset path, CDN, WebJar, npm build, or internal asset pipeline.

## Runtime requirements by component

All Kotlin wrappers render HTML on the server. Every component expects `/mtds/styles.css` to be served by the application.

Components that are CSS-only or rely only on native browser behavior can render without `/mtds/index.iife.js`: `Alert`, `Avatar`, `Badge`, `Button`, `ButtonLink`, `Card`, `Chip`, `Details`, `Divider`, `Fieldset`, `FileUpload`, `Flex`, `Grid`, `Heading`, `Input`, `Link`, `Print`, `Skeleton`, `Spinner`, `Steps`, `Tag`, `Typography`, and similar generated native-tag wrappers. Some of these may still need application-provided IDs, form names, or submit handling.

Components or helpers that depend on design-system custom elements, observers, or runtime behavior require `/mtds/index.iife.js`: `App`, `Breadcrumbs`, `Chart`, `Dialog` command/popover interactions, `ErrorSummary`, `Logo` observer behavior, `Popover`, `Progress`/`u-progress`, `Table` responsive/sort observers, `Tabs`, `Toast`, `Tooltip` (`data-tooltip`), and dynamic `Validation` behavior.

Atlas components require `/mtds/atlas.iife.js` instead of only the main bundle. Atlas also depends on Leaflet behavior, map tiles, and external services such as Matgeo where those layers are used; make network, privacy, and availability requirements explicit in the consuming application.

## Generated code

Component contracts are checked-in TS source input:

```txt
designsystem/button/button.contract.ts
designsystem/alert/alert.contract.ts
```

`npm run build` runs `npm run build:metadata`, which validates those contracts and emits generated JSON metadata:

```txt
mtds/component-metadata/button.component.json
mtds/component-metadata/alert.component.json
```

`./gradlew -p kotlin generateMtdsKotlin` and the normal Kotlin `build` task consume the generated JSON from `mtds/component-metadata`. Run `npm run build` first if that directory is missing or stale.

Generated Kotlin sources are written to:

```txt
kotlin/build/generated/mtds/io/mattilsynet/design/...
```

Generated resources are written to:

```txt
kotlin/build/generated/resources/main/...
```

Do not edit generated Kotlin, generated resources, or generated `mtds/component-metadata/*.component.json` files by hand. Update the checked-in TS component contracts instead.

The build validates that each metadata `className` exists in `mtds/styles.json`.

## Deferred components

Components with custom browser behavior, advanced accessibility contracts, or complex nested structures are intentionally deferred until their server-rendered HTML contract is explicit. Examples include `Chart`, `Dialog`, `Popover`, `Toast`, `Tooltip`, `Table`, `Tabs`, `FileUpload`, `Combobox`, `Atlas`, `Law`, `App`, `Pagination`, `Breadcrumbs`, `Details`, `HelpText`, `ToggleGroup`, `Steps`, `Avatar`, and `Progress`.

## Troubleshooting

### Missing `mtds/styles.json`, `mtds/styles.css`, or `mtds/component-metadata`

Run the root npm build first:

```sh
npm run build
```

### Missing npm packages during `npm run build`

Install dependencies from the lockfile:

```sh
npm install
```

### HTMX attributes render but nothing happens in the browser

This library only provides `hx-*` attribute helpers. Add the HTMX runtime in your application, for example:

```html
<script src="https://unpkg.com/htmx.org@2.0.4"></script>
```
